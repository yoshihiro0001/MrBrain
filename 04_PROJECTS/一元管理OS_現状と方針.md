# 一元管理OS 現状と方針

## 目的
経理アプリ単体ではなく、JOY・孝行・税務・経理・スタッフ・契約・設備を一元管理するOSを作る。

目的は、脳をすっきりさせ、同じ説明をAIに何度も繰り返さず、記録を安全に残しながら、必要な情報にすぐたどり着ける状態を作ること。

上位概念として、[[07_SYSTEM/エネルギー循環OS憲法|エネルギー循環OS憲法]] に従う。

一元管理OSは、エネルギー循環OSの実務レイヤーである。

## 今の状態
情報と開発が同時に増えすぎて、何を整理すればいいか分からなくなり、開発が止まりやすい状態。

ただし、これは失敗ではない。今まで断片的に作ってきたものが、実は一つの大きなOS構想につながっていることが見えてきた段階。

## これまでの流れ
1. JOY管理シートを作り、ホテル運営の情報を見える化した。
2. 孝行の申告のため、決算書や経費情報をAIに渡し、スプレッドシートで申告システムを作った。
3. `孝行_第40期申告システム_確定` を完成させ、全力法人税への入力、e-Tax、郵送申告まで進めた。
4. 申告に時間がかかりすぎたため、次回以降のために `孝行_申告システム_汎用版2` を作った。
5. 経費漏れ、レシート入力、証憑管理、スタッフ入力の面倒さに気づき、アプリ開発を進めた。
6. その結果、アプリは単なる経理アプリではなく、会社切替・スタッフ権限・承認・証憑・仕訳連携を含むOSの入口になってきた。
7. 同じ説明をAIに何度もしていることに気づき、MrBrainを最初に整える必要があると判断した。

## 混乱の本質
混乱の原因は、整理能力がないことではない。

原因は、次の3つが同時に動いていること。

1. 実務の整理
   - JOY
   - 孝行
   - 契約
   - 税務
   - スタッフ
   - 設備

2. データの整理
   - スプレッドシート
   - 申告システム
   - 経費入力
   - 証憑画像
   - 仕訳帳

3. アプリ開発
   - OCR
   - スタッフ入力
   - 承認
   - 会社切替
   - 権限
   - データベース

この3つを同時に完璧にしようとすると止まりやすい。

## 最重要判断
集約先はアプリではない。

集約先はMrBrain。

さらに上位では、目的は管理ではなく循環である。

アプリは、MrBrainで整理された設計思想とルールを実行する道具。
スプレッドシートは、現在のデータと計算の置き場。
GitHubは、開発コードと履歴の置き場。

## 役割分担
| 置き場所 | 役割 | 入れるもの |
|---|---|---|
| MrBrain | 頭の中の整理、索引、判断、方針 | 目的、現状、ルール、リンク、決定ログ |
| Googleスプレッドシート | 現在動いている業務データ | 売上、経費、契約、申告表、管理表 |
| 開発アプリ | 入力と操作のUI | レシート入力、承認、会社切替、スタッフ入力 |
| GitHub | コードと変更履歴 | Reactアプリ、設計ファイル、Issue、PR |
| Google Drive | 原本・証憑・画像の保管 | レシート画像、契約書、申告資料 |

## 今後の判断基準
- 迷ったら、まずMrBrainに要約を書く。
- スプレッドシートを増やす前に、どの正本にするか決める。
- アプリに機能を足す前に、MrBrainで「目的」「入力」「出力」「使う人」を書く。
- 税務・契約・給与・個人情報は、原本ではなく索引と保管場所だけを書く。
- 完璧に統合する前に、まず「正本」を決める。
