# JOY 既存システム再現OS Blueprint

## 重要な修正
このBlueprintの目的は、既存システムのUIをそのまま再現することではない。

本当に必要なのは、現場動作や信号が、PC上でどんな処理に変換され、料金、売上、DB保存へ流れているかを洗い出すことである。

正本としては、まず `JOY_信号_PC処理対応Blueprint.md` を優先する。

```md
現場動作・信号
↓
PC上の処理・反応
↓
料金計算
↓
売上イベント
↓
DB保存
↓
自社OSの見やすいUI
```

既存UIは参考資料であり、完成形ではない。

## 位置づけ
これは、月次承認OSや税務出力の前に作る、JOYの売上源流コンポーネントのBlueprintである。

最初のGoalは、月次承認ではない。

最初のGoalは、JOY既存システムと同じように、部屋状態、休憩、宿泊、延長、清掃、ドア、冷蔵庫売上などの信号を受け取り、好きなUIで再現できるシステムを作ることである。

```md
信号取得
↓
イベント化
↓
部屋状態管理
↓
料金計算
↓
売上発生
↓
DB保存
↓
月次承認へ接続
```

## 1. 本当のGoal
202号室を代表点として、既存PCへ入っている信号・入力経路を推定し、自社OSで同じ部屋状態・料金・売上イベントを再現できる設計を作る。

## 2. Success Condition
- 202号室の操作が、既存PC上でどの状態変化・料金変化を起こすか分かる
- 既存PCへ入っている入力形式の有力候補が分かる
- 自社OSイベントへ変換する対応表が作れる
- 202号室だけで、信号取得から売上発生までの最小デモ設計に進める

## 3. Hypothesis
現時点の有力仮説は次の順番で確認する。

1. 既存PCはUSB HIDまたはキーボード相当の入力を受けている
2. USB HIDでなければ、COM / シリアル入力の可能性がある
3. それでもなければ、LAN、ローカルログ、帳票、画面表示の順に確認する

## 4. 必要コンポーネント
| コンポーネント | 役割 |
|---|---|
| Signal Intake | 既存システムから来る入力経路を読む候補を管理する |
| Event Mapper | 入力を `room_event` へ変換する |
| Room State Engine | 部屋状態を変える |
| Rate Rule Engine | 休憩、宿泊、延長、冷蔵庫売上を料金へ変換する |
| Sales Event | 売上発生イベントを作る |
| Room Master | 14部屋の部屋マスタを持つ |
| Rate Master | 料金表、時間帯、延長、商品料金を持つ |
| Existing System Map | 既存PC反応と自社OSイベントの対応表 |
| UI Dashboard | 部屋状態、料金、イベントを見やすく表示する |
| Event Log | すべての操作と推定を記録する |

## 10. UI方向性
目的は、既存システムを真似ることではない。

目的は、好きなデザインで、部屋状態と売上源流が一瞬で分かる画面にすること。

最小画面:
- 部屋一覧ボード
- 202号室詳細
- イベントログ
- 料金内訳
- 既存PC対応表

UIの方向性:
- 部屋状態がひと目で分かる
- 色を使いすぎない
- 202号室だけでも完成度を高くする
- スマホやiPadで見ても確認しやすい
- 既存PC画面より気持ちよく、でも現場で迷わない

## 11. 202号室だけで検証する最小テスト
まだ配線、常駐ソフト、自動取得はしない。

やること:
1. PC裏、入力機器、キーボード、デバイスマネージャーを撮る
2. 202号室で1つだけ操作する
3. 既存PC画面、音声、料金、伝票、部屋状態の変化を記録する
4. 自社OSイベント候補へ変換する
5. 料金計算に必要な情報が足りるか確認する

## 12. 足りない情報
- 実際の入力方式
- 既存PCソフト名
- 既存ソフトの保存先
- 料金表の正確な内容
- 冷蔵庫売上の反映方法
- 取消、修正の操作方法
- 支払い方法の扱い
- 既存データの出力可否

最初のコードは、自動取得ではなく、202号室イベントを手入力で再現できる最小UIから始める。

## 14. Blueprint Review
- 源流である信号、入力、部屋状態、料金計算を最初に見ているか
- 月次承認や税務出力へ先に進んでいないか
- 202号室だけで最小検証できるか
- 既存システムに触らず、読み取り前提で進めているか
- 料金計算と売上発生までつながっているか
- 好きなUIで再現する余地が残っているか
- 足りない情報が明確か
- 次のコード化順が、源流から下流になっているか
