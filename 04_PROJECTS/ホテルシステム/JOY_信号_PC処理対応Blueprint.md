# JOY 信号 PC処理対応Blueprint

## 位置づけ
これは、JOYの既存システムを見た目ごと再現するためのBlueprintではない。

目的は、現場動作や信号が、PC上でどんな処理に変換され、料金計算、売上イベント、DB保存へ流れているかを対応表にすることである。

```md
現場動作・信号
↓
PC上の処理・反応
↓
料金計算
↓
売上イベント
↓
DB保存
↓
自社OSの見やすいUI
```

UIは既存システムを真似ない。
自社OSでは、部屋状態、料金、売上源流、確認不足が一瞬で分かるシンプルなUIへ再設計する。

## 1. 本当のGoal
JOYの現場操作が、既存PC上でどんな処理に変換されているかを洗い出し、自社OSで同じ売上源流を扱えるイベント構造を作る。

## 2. Success Condition
- 202号室の代表操作について、現場動作、PC上の反応、料金変化、売上イベントの対応表が作れる
- 既存PCのUIを真似ずに、自社OS用のイベント、状態、料金、DB構造へ変換できる
- 自動取得へ進む前に、手入力でも202号室の売上源流を再現できる設計になる
- 月次承認、経理、税務へ流す前の売上発生ロジックが見える

## 3. 洗い出すべき現場動作・信号一覧
最初は202号室だけで確認する。

| 分類 | 現場動作・信号 | 確認したいこと |
|---|---|---|
| 入室 | 客室利用開始 | PC上で利用開始、時間開始、部屋状態変更が起きるか |
| 休憩 | 休憩ボタン、休憩扱い | 休憩料金と制限時間がどう決まるか |
| 宿泊 | 宿泊ボタン、宿泊扱い | 宿泊料金、開始時刻、切替条件がどう決まるか |
| 延長 | 時間超過、延長操作 | 延長料金が自動か手動か |
| 清掃 | 清掃中、清掃完了 | 売上締め後の部屋状態がどう変わるか |
| ドア | ドアオープン | 音声、画面表示、状態変化の有無 |
| 冷蔵庫 | 商品利用、補充、取消 | 商品名、数量、金額が伝票へどう入るか |
| 取消 | 誤操作取消 | 伝票、料金、状態、ログがどう戻るか |
| 修正 | 金額、区分、時間の修正 | 修正履歴が残るか |
| 精算 | 支払い、退室、締め | 売上確定のタイミングはどこか |

## 4. PC上で見るべき処理・反応
内部処理を直接読む前に、まず安全に見える反応を観測する。

| 観測対象 | 見ること |
|---|---|
| 画面表示 | 部屋番号、状態、経過時間、料金、警告、色の変化 |
| 音声 | 「202 ドアオープン」などの音声タイミング |
| 伝票 | 利用区分、商品、延長、合計金額、支払い方法 |
| ログ | 操作履歴、時間履歴、エラー履歴があるか |
| 帳票 | 日計、売上集計、部屋別集計、商品別集計 |
| ファイル | CSV、バックアップ、ローカルDB、設定ファイル |
| 入力経路 | USB HID、キーボード、COM、LAN、専用入力装置 |

## 6. 必要な料金ルール
料金計算を再現するために、次を確認する。

- 部屋番号ごとの料金差
- 休憩料金
- 宿泊料金
- 延長料金
- 時間帯ルール
- 曜日、平日、休日の違い
- 冷蔵庫商品料金
- 割引、取消、修正の扱い
- 支払い方法
- 消費税の扱い
- 売上確定タイミング

料金表は、最初から全ルールを完全化しない。
202号室の1パターンで、料金がどう発生するかを先に確認する。

## 7. 部屋状態の状態遷移
最初の状態遷移候補:

```md
空室
↓
入室
↓
休憩 または 宿泊
↓
延長あり または 延長なし
↓
退室
↓
清掃中
↓
清掃完了
↓
空室
```

ドアオープンや冷蔵庫売上は、状態そのものではなく、状態に紐づくイベントとして扱う。

## 8. 追加処理
### 冷蔵庫売上
- 商品コードまたは商品名
- 数量
- 金額
- 発生時刻
- 部屋番号
- 伝票反映タイミング
- 取消可否

### 延長
- 自動延長か手動延長か
- 延長単位
- 延長料金
- 画面警告の有無
- 売上反映タイミング

### 取消・修正
- 誰が操作したか
- 何を戻したか
- 元イベントが残るか
- 差額処理か、上書きか
- 税務・売上集計上の扱い

## 9. AIが自動計算できる部分
情報が揃えばAIまたは自社OSで自動化しやすい部分:

- 経過時間
- 休憩、宿泊、延長の料金候補
- 冷蔵庫商品合計
- 部屋状態の履歴表示
- 売上イベントの候補生成
- 未確認イベントの抽出
- 料金表と実売上の差分検知

## 10. 人間が確認すべき部分
自動化しすぎない部分:

- 料金表が正しいか
- 取消、修正、割引の理由
- 現金差異
- カード決済との不一致
- 税務上の扱いが不明なもの
- 既存システムの保守や安全性に関わる操作
- 配線、制御、既存DB書き込みに関わる判断

## 12. UI方針
既存UIは完成形ではない。

自社OSのUIは、次を優先する。

- 部屋状態がひと目で分かる
- 料金の発生理由が分かる
- どの信号から来たイベントか分かる
- 未確認、要確認、承認済みが分かる
- 202号室だけでも気持ちよく使える
- スマホ、iPad、PCで確認しやすい

最小画面:

1. 部屋一覧
2. 202号室詳細
3. イベント履歴
4. 料金内訳
5. 信号 PC処理対応表

## 13. コード生成へ進む順番
情報が揃ったら、次の順番で進む。

1. 202号室の手入力イベント登録
2. room_event DB
3. Room State Engine
4. Rate Rule Engine
5. sales_event DB
6. 202号室詳細UI
7. 部屋一覧UI
8. 信号 PC処理対応表UI
9. 未確認イベント表示
10. 読み取りレイヤー接続候補の実装検討

最初の実装は、信号の自動取得ではなく、観測したPC処理を手入力でイベント化できる最小UIから始める。

## 14. 最初の1回で集める情報
まずはこれだけでよい。

ホテル信号調査では、写真、型番、スクショ、画面表示などを最小確認情報として扱う。
これはホテルProject固有の確認行動であり、汎用的な思考原理そのものではない。

```md
202号室で1つ操作した動画または連続写真
操作前の既存PC画面
操作後の既存PC画面
料金表
伝票または料金表示
PC裏写真
入力機器写真
キーボード型番
デバイスマネージャー
```

この情報だけで、PC上の処理と自社OSイベント構造を7割まで推定する。

- 既存UI再現が目的になっていないか

- 202号室だけで検証できるか
