# JOY 信号取得レイヤー判定Blueprint

## 目的
JOYの現場操作から発生する信号を、自社OSがどのレイヤーで読むのが最高効率かを判定する。

目的は、アールメックス / USEN既存システムと深く統合することではない。

目的は、次の最小構造を理解し、自社OSへ接続できる場所を見つけることである。

```md
現場操作
↓
電気信号
↓
入力機器
↓
PCが読める形式
↓
既存ソフトが状態・料金に変換
```

自社OSで目指す流れ:

```md
現場操作
↓
電気信号
↓
入力機器
↓
PCが読める形式
↓
自社OSがイベントとして理解
↓
DB保存
↓
画面表示
↓
売上・清掃・経理へ接続
```

## このBlueprintが必要な理由
これまでの調査は安全性を重視していた。

それは正しい。

ただし、ユーザーが本当に知りたい上流は、次の問いである。

```md
信号をアプリが読むには、どのレイヤーで拾うのが最高効率か。
```

いきなり電気信号を読むのではなく、まずはPCに入った後の入力形式を特定する。

## ユーザーの思考特性
このホテル信号調査では、現場操作を「信号 → 入力 → 変換 → 出力 → 循環」の最小構造へ分解し、最短で実装可能な取得レイヤーを見つける。

この思考は、ホテルシステムだけでなく、経理、投資、人体、AI連携、SNSにも応用できる可能性がある。

ただし、現時点では新しい原理として昇格させず、Observation / Patternとして扱う。

| 入力機器の出力 | ハブや入力装置の出力を読む | 中 | 中〜低 | 高 | 中 | 高 | 高 |

| USB HID | USB入力機器として読む | 中 | 中 | 高 | 中 | 高 | 高 |
| シリアル通信 | COMポート経由のデータを読む | 中 | 中〜低 | 高 | 中 | 高 | 高 |

## 現時点の最高効率候補
現時点で最も確認価値が高い候補は、次の3つである。

```md
1. キーボード入力
2. USB HID
3. シリアル通信
```

理由:
- ホテル現場の操作は、ボタンや入力装置からPCへ何らかのイベントとして入っている可能性が高い
- PCに入った後の形式を特定できれば、自社OSがイベントとして受け取る設計に近づく
- 電気信号そのものより、入力形式を読む方が安全で実装に近い

ただし、最初に触るのは信号ではない。

ホテル信号調査では、写真、型番、スクショ、画面表示などを最小確認情報として扱う。
これはこのProjectで信号取得レイヤーを推定するための具体的な確認行動であり、汎用原理ではない。

まず見るもの:

```md
キーボード写真
キーボード型番
ハブ / 入力機器の写真
接続端子の写真
Windowsデバイスマネージャー
既存ソフト画面
操作時のPC表示
```

## 推定ルート

### キーボード入力の可能性が高いサイン
- 専用キーボードやボタン盤がPCに接続されている
- USBキーボードとして認識されている
- 操作時に入力欄や画面が即時反応する
- Windows上でHID Keyboard Deviceが複数ある

見る場所:
```md
Windowsデバイスマネージャー
キーボード項目
USB接続機器
専用キーボード型番
```

### USB HIDの可能性が高いサイン
- USB接続の入力ボックス、変換器、専用機器がある
- デバイスマネージャーにHID準拠デバイスがある
- キーボードではないが入力装置として認識されている

見る場所:
```md
ヒューマンインターフェイスデバイス
USBコントローラー
接続端子
機器型番
```

### シリアル通信の可能性が高いサイン
- RS-232C、COM、D-sub 9ピンのような端子がある
- デバイスマネージャーにCOMポートがある
- 古い業務機器、制御盤、変換器を経由している

見る場所:
```md
ポート（COMとLPT）
USB-Serial変換器
機器型番
ケーブル形状
```

### LAN通信の可能性が高いサイン
- 既存PCと制御機器がLANケーブルでつながっている
- 専用機器にIPアドレス設定がある
- 画面更新がネットワーク越しに行われている可能性がある

見る場所:
```md
LANケーブル
ネットワークアダプター
機器のIP設定画面
ルーター / ハブ
```

### ローカルDB / ログの可能性が高いサイン
- 既存ソフトの保存フォルダがある
- 日報、伝票、状態履歴がPC内に残る
- 古いデータが2年分程度保存されている

見る場所:
```md
既存ソフトの保存先
バックアップフォルダ
ログフォルダ
データベースファイルらしきもの
日報 / 月報出力
```

### 画面表示読み取りが現実的なサイン
- 入力経路が不明でも画面表示は安定している
- 「202 ドアオープン」のような明確なテキストが出る
- 部屋状態、料金、清掃状態が画面で確認できる

見る場所:
```md
既存PC画面
モニター表示
音声案内
画面上の文字
```

## 最小確認情報
以下だけで、7割程度の推定を目指す。

```md
1. 専用キーボードの写真
2. キーボード型番
3. ハブ / 入力機器の写真
4. 接続端子の写真
5. Windowsデバイスマネージャーのスクショ
6. 既存ソフト画面のスクショ
7. 202号室で操作した時のPC表示
8. 操作時の音声・伝票・料金変化メモ
```


## 最終ゴール
既存システムは併走させたまま、自社OSも同じイベントを受け取る。

PC画面を切り替えると、自社OS側でも次が見える状態にする。

```md
部屋状態
売上
清掃状態
冷蔵庫イベント
利用時間
料金変化
```

最初は制御しない。

まずは、読み取り専用でイベントとして理解する。

## Blueprint Review判定
判定: 調査Blueprintとして実行候補。配線・実装・自動取得は保留。

理由:
- 本質が「既存統合」ではなく「信号取得レイヤー判定」に整理された
- まずPCが読める形式を特定すれば、自社OS接続の最短ルートが見える
- 写真、型番、デバイスマネージャー、画面表示だけで初期仮説を立てられる
- 既存システムに書き込まず、安全に調査できる
