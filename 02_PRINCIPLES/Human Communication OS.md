# Human Communication OS

## 位置づけ
これは、樫原義大OSにおける人間理解に基づくコミュニケーション原則である。

単なるメール文例ではない。

接客、クレーム対応、経営者としての依頼、スタッフ教育、家族との会話、AIへの指示に共通する「人と話す順番」を定義する。

## このファイルが必要な理由
毎回ゼロから文章を考えると疲れる。

しかし、原則とテンプレートがあれば、感情的になりやすい場面でも、相手を理解し、安心を作り、必要な確認や解決へ進められる。

今後、返信文生成スキルやCommunication Agentを作るときの正本にする。

## 中核原則
人は問題解決の前に理解されたい。

人は、まず理解されたい。

理解された後に、解決を受け入れやすくなる。

## エネルギー循環OSとの接続
[[02_PRINCIPLES/エネルギー循環OS|エネルギー循環OS]] では、人を欠点や弱点ではなく、部品と接続と循環で見る。

Human Communication OSは、その接続の前段階を担当する。

安心がない状態で部品を渡しても、相手は受け取れない。

だから順番は次になる。

```md
理解
↓
安心
↓
接続
↓
循環
↓
行動
```

返信、接客、クレーム対応、スタッフ指導では、正しい部品を渡す前に、相手が受け取れる状態を作る。

## なぜトラブルになるのか
多くの人は、次の順番で会話しようとする。

```md
問題
↓
解決
```

しかし人間は、実際には次の順番で動く。

```md
感情
↓
理解
↓
安心
↓
解決
```

問題だけを見て解決に進むと、相手は「分かってもらえていない」と感じる。

感情を理解してから確認へ進むと、同じ内容でも印象が大きく変わる。

## 発見の背景
今回の歯科で感じた違和感から、次のことが見えた。

```md
理解されないと、人は不満を感じる。
```

これは嫌な体験で終わらせるものではなく、HOTEL JOY運営、スタッフ教育、家族との会話、AIへの指示へ展開できる重要な発見である。

## 失敗パターン
お客様:
```md
Wi-Fiが繋がりません。
```

運営:
```md
部屋番号を教えてください。
```

運営側は解決しようとしている。

しかし、お客様には「困っていることを理解してもらえた」という感覚が残りにくい。

## 成功パターン
お客様:
```md
Wi-Fiが繋がりません。
```

運営:
```md
ご不便をおかけし申し訳ございません。

お困りの状況を改善したいので、部屋番号と時間帯を教えていただけますでしょうか。
```

聞いている内容は同じ。

しかし印象は全く違う。

## 基本ルール

### Rule 1 共感を先に置く
解決の前に理解。

悪い例:
```md
詳細を教えてください。
```

良い例:
```md
ご不便をおかけし申し訳ございません。
状況を確認させてください。
```

### Rule 2 感情と事実を分ける
まず感情を受け止める。

その後、事実確認に進む。

```md
不安
怒り
悲しみ
焦り
↓
共感
↓
事実確認
```

### Rule 3 説明より理解
人は説明されたいのではない。

理解されたい。

悪い例:
```md
個人差があります。
規約ではこうなっています。
仕様です。
```

良い例:
```md
ご不安なお気持ちは理解できます。
その上で、現状をご説明いたします。
```

### Rule 4 正しさより安心
正論は相手を納得させない。

安心が納得を生む。

### Rule 5 文章より行動
言葉が完璧でなくても、次の行動があれば誠実さは伝わる。

- 確認した
- 調査した
- 修正した
- 連絡した
- 再発防止した

言葉だけ丁寧で何もしない方が問題。

### Rule 6 役割ごとに順番を変える
同じ人間理解でも、場面によって順番を変える。

経営者モード:
```md
結論
↓
理由
↓
質問
```

接客モード:
```md
共感
↓
確認
↓
解決
```

クレーム対応モード:
```md
共感
↓
謝罪
↓
確認
↓
解決
```

家族モード:
```md
理解
↓
共感
↓
提案
```

AI指示モード:
```md
ゴール
↓
制約
↓
成果物
↓
判断基準
```

## HOTEL JOY 原則
お客様は、問題だけを持って来館するのではない。

感情を持って来館する。

運営は問題だけを見るな。

感情を見る。

感情が整うと、問題解決は半分終わっている。

## 樫原義大 原則
相手を説得しない。

理解する。

理解された人は、自然に協力してくれる。

優先する順番:
```md
説明の前に理解
解決の前に安心
正しさの前に信頼
```

## Communication Agent

### 役割
ユーザーの文章を読み、本音を1行に圧縮し、最適なテンプレートへ当てはめて返信を生成する。

### 入力
- 思ったこと
- 相手
- 目的
- 感情
- 絶対に避けたいこと
- 望む結果

### 出力
- 本音1行
- 相手の感情仮説
- 使うテンプレート
- 返信案
- 強さ調整
- 注意点

### 圧縮の考え方
ユーザーは思ったことを100書いてよい。

Communication Agentが、100を3に圧縮する。

```md
長い感情・事情
↓
本音1行
↓
相手に伝わる返信
```

## 判断基準との結びつき
- シンプルを優先:
  - まず共感、確認、解決の順番だけ守る。
- 長期運用を優先:
  - テンプレート化して毎回考えない。
- 人力を減らす:
  - Communication Agentで長文を返信案へ圧縮する。
- 同じ説明を繰り返さない:
  - 返信原則をこのファイルに固定する。
- 情報は構造化して保存する:
  - 原理、判断基準、テンプレート、例文、AIプロンプトに分ける。
- エネルギー循環:
  - 理解されない停滞を減らし、安心と協力の循環を作る。

## 注意
このOSは、相手に迎合するためのものではない。

相手を理解し、自分の目的も守るための順番である。

謝ることと、すべての責任を認めることは別。

共感することと、相手の主張を全面的に認めることも別。
