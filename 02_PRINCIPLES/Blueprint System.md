# Blueprint System

## 位置づけ
Blueprint Systemは、思想や発見を実装前の設計図へ変換するためのOSである。

ブランド名は `Flow Builder`。

内部構造名は `Blueprint System`。

Blueprint Systemの上流には、[[02_PRINCIPLES/Blueprint Engine|Blueprint Engine]] を置く。

Blueprint Engineは、設計図を作る前に必ず通る思考アルゴリズムである。

現在の上位方針:
Blueprint Systemは、単独で実装へ進むものではない。
現在は [[07_SYSTEM/MrBrain Runtime Loop|MrBrain Runtime Loop]] の Route / Shape の中で使う設計部品であり、実装前には Verify として [[07_AGENTS/Blueprint Reviewer|Blueprint Review]] を通す。

```md
憲法
↓
Blueprint Engine
↓
Blueprint System
↓
Blueprint
↓
実装
↓
運用
↓
改善
```

## このファイルが必要な理由
ユーザーは、情報を集めることが苦手なのではない。

情報をどう実装へ接続するかで止まりやすい。

止まっている場所は、思考ではない。

開始点である。

だから、次に必要なのは設計図。

## 中核原則
思想の次に、必ずBlueprintを通す。

```md
思想
↓
Blueprint Engine
↓
Blueprint
↓
実装
↓
運用
```

設計図を飛ばすと、途中で迷い、複雑化し、表示崩れや実装トラブルが起きやすい。

## 目的
- 何から相談すればよいかを明確にする
- 実装前に目的、現在地、完成形、必要部品をそろえる
- AIにも人間にも分かりやすい設計図を作る
- 複雑化を防ぐ
- 変更しても壊れにくい形にする
- 実装後に運用、改善、保存へ戻せるようにする

## 現在最適性チェック
Blueprintを作る前に、必ず次を確認する。

```md
現在の前提は何か。
もっと良い構造や技術は出ていないか。
この設計はシンプルさ、保守性、拡張性、循環性の観点で今も最適か。
より少ない部品で同じ目的を達成できないか。
```

ここでいう「最新」は年号で固定しない。

作業時点で、公式情報、一次情報、実際の利用可能性、運用コスト、安全性、AI連携性を確認したうえで、ユーザーにとって最も価値が高い構成を選ぶ。

## スキル化Blueprint
CodexやAIスキルを作る場合。

```md
繰り返し作業
↓
手順化
↓
入力
↓
出力
↓
禁止事項
↓
テスト
↓
スキル化
```

見る部品:
- 何を毎回繰り返しているか
- どの入力が必要か
- どの出力が必要か
- 守るルール
- 使ってはいけない情報
- 失敗例
- 成功例

判断:
- 3回以上同じ流れを使ったらスキル化候補
- 1回だけならテンプレートで十分
- 外部送信や個人情報が絡む場合は慎重にする

## Blueprint Agentとの関係
[[07_AGENTS/Blueprint Agent]] は、このBlueprint Systemを使って設計図を作るAgentである。

ユーザーが「経理を便利にしたい」「Booking.comを効率化したい」「このアプリを作りたい」と言ったら、Blueprint Agentは設計図を先に作る。

## Flow Builderの一文定義
Flow Builderとは、発見や目的を、迷わず実装へ進めるための設計図に変換するOSである。

## 判断基準
- 目的が一文で言えるか
- 現在地が見えているか
- 完成形が見えているか
- 接続する部品が列挙されているか
- 必要な情報が分かっているか
- 使うAIやツールが決まっているか
- 実装方法が最小構成になっているか
- テスト方法があるか
- 運用後にどこへ保存するか決まっているか
