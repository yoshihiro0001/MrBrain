# Blueprint Engine

## 位置づけ
Blueprint Engineは、MrBrainの最上位憲法から、毎回ぶれにくい設計図を生み出すための設計アルゴリズムである。

Blueprintは成果物。

Blueprint Engineは、成果物を作る前に必ず通る思考の順番。

ただし、Blueprint Engineは最初に呼び出すものではない。

まず `07_SYSTEM/MrBrain Runtime Loop.md` のThinking Engineで、対象、本当のGoal、Hypothesis、Success Condition、Source First Check、対象世界、世界のルール、目的、本質、Root Structure、Representative Point、Leverage Point、Minimal Information Set、最小構造、不足情報、最短ルートを確認し、Blueprintが必要と判断された場合だけ使う。

Blueprintは思考の出発点ではない。

必要な場合の設計成果物である。

重要なBlueprintは、作って終わりにしない。

交換可能なComponentにできるか、自然に循環するFlowへ接続できるか、他分野へ再利用できるかを確認する。

また、価値の低い反復判断を圧縮し、価値の高い思考にエネルギーを戻せるかを確認する。

標準化、部品化、OS化、判断圧縮は目的ではなく、循環を止める摩擦を消すための実装パターンとして扱う。

最後に、ユーザーが読む量、考える量、確認する量、動く量をさらに減らせないか確認する。

## このファイルが必要な理由
MrBrainには、憲法、判断基準、接続理論、Blueprintテンプレートがある。

しかし、今のままだとAIが「Blueprintを作って」と言われた時に、その時々で違う順番で考える可能性がある。

必要なのは、単なる設計図ではなく、最高の設計図を毎回生み出すエンジンである。

## 最上位憲法との関係
Blueprint Engineは、MrBrainの最上位憲法から導かれる。

```md
あらゆる現象は、接続・循環・適応から成るシステムとして理解できる。
改善とは、滞った接続を見つけ、最小の介入で循環を回復し、適応を促すことである。
```

したがって、Blueprint Engineは次の階層で考える。

```md
存在
↓
システム
↓
接続
↓
循環
↓
適応
↓
成長
```

## 中核思想
ユーザーが欲しいのは、設計図そのものではない。

欲しいのは、どんな領域でも循環する設計図を作れる仕組みである。

```md
世界の原理（憲法）
↓
Blueprint Engine
↓
Blueprint
↓
Blueprint Review
↓
Blueprint更新
↓
承認
↓
Coding Agent
↓
テスト
↓
運用
↓
結果測定
↓
Evolution Loop
↓
Blueprint Engine改善
↓
次の実装
```

コードは最後に置く。

ホテルシステム、経理、SNS、人体、投資は目的ではなく、Blueprint Engineを検証する実験場である。

本当に育てるものは、より良いBlueprintを生み出し、そのBlueprint自身も学習・進化し続けるOSである。

## 人とAIが同じOSを育てる循環
Blueprint Engineは、AIに判断を丸投げするためのものではない。

AIが設計し、人間が承認し、実行結果からEngineが改善されることで、人とAIが同じOSを育てる。

```md
世界の原理
↓
私の判断基準
↓
Blueprint Engine
↓
AIが設計する
↓
私は承認する
↓
AIが実行する
↓
結果を測定する
↓
Blueprint Engineを改善する
↓
次はさらに良い設計になる
```

```md
AIが学ぶ
↓
私が学ぶ
↓
OSが進化する
↓
またAIが学ぶ
```

## Evolution Loop
Blueprint Engineは、作って終わりではない。

一度作ったBlueprintを現実で試し、結果を測定し、学びを戻すことで、次のBlueprintを良くする。

```md
接続
↓
循環
↓
適応
↓
学習
↓
進化
↓
より良い接続
```

Evolution Loopで戻すもの:
- どこで止まったか
- どの接続が足りなかったか
- どの入力が不要だったか
- どの自動化が有効だったか
- どの判断基準がズレたか
- 次のBlueprint Engineに追加すべき問い

## Blueprint Review
Blueprint完成後、すぐに実装へ進まない。

実装前に必ず [[07_AGENTS/Blueprint Reviewer|Blueprint Reviewer]] を通す。

```md
Blueprint完成
↓
Blueprint Reviewer
↓
改善案
↓
Blueprint更新
↓
承認
↓
Coding Agent
↓
コード生成
```

Blueprint Reviewで確認すること:
- もっとシンプルにできないか
- 接続漏れはないか
- 循環が止まる場所はないか
- 将来拡張しやすいか
- 他分野にも再利用できるか
- 作業時点の技術水準で本当に最適か
- 私の判断原理に沿っているか
- 個人情報、税務情報、契約情報を安全に扱えているか
- テスト方法と戻し方があるか

レビューを通過したBlueprintだけを、Coding AgentやCodexの実装へ渡す。

Blueprint Review自体もEvolution Loopの対象にする。

レビューで見落としがあった場合は、レビュー項目そのものを改善する。

## 固定と進化の分離
MrBrainでは、次のように分ける。

```md
憲法は固定する
Blueprint Engineは常に作業時点の最新知見で最適化する
```

ここでいう「最新」は、特定の年号ではない。

レビューや実装を行う時点で、公式情報、一次情報、実際の利用可能性、運用コスト、安全性、AI連携性を確認したうえで、ユーザーにとって最も価値が高い構成を意味する。

## 承認レベル
自動化は、いきなり完全自動にしない。

リスクと再現性に応じて、承認レベルを上げる。

| Level | 状態 | 使う場面 |
|---|---|---|
| Level 1 | AIが提案し、私が細かく修正する | 新しい領域、判断基準がまだ固まっていない作業 |
| Level 2 | AIが提案し、私はOK / NGだけ判断する | 型が見え始めた作業、低〜中リスクの反復作業 |
| Level 3 | AIが95%以上の確信を持つ低リスク案件だけ自動実行する | 失敗しても戻せる作業、明確なルールがある作業 |
| Level 4 | 小さなリスクの仕事は自動実行し、大きな意思決定だけ私が確認する | 十分に検証済みで、ログと復元方法がある作業 |

承認レベルを上げる条件:
- 実行ログが残る
- 失敗時に戻せる
- 個人情報、税務情報、契約情報の扱いが安全
- 同じ作業を複数回成功している
- 私の判断基準とズレていない
- 結果をOperation Logへ戻せる

承認レベルを上げない条件:
- 税務、契約、給与、個人情報に大きく関わる
- 金銭的影響が大きい
- 外部公開される
- 失敗時に戻しにくい
- 判断基準がまだ曖昧
- 私の違和感が残っている

## 回答前に必ず通る10段階
AIは、設計、文章、コード、返信、分析を出す前に、内部でこの順番を通る。

```md
1. 目的
2. 入力
3. 出力
4. 構成要素
5. 接続
6. 循環
7. 停滞
8. 最小介入
9. 将来の拡張性
10. 実行
```

## 詳細

### 1. 目的
何の循環を良くしたいのかを一文で定義する。

例:
- お客様との信頼循環を回復する
- 経理入力から申告数字までの循環を整える
- スタッフの行動と役割の循環を良くする
- AIとの説明ループを減らす

### 2. 入力
何が入ってくるかを見る。

例:
- 問い合わせ文
- レシート
- 予約情報
- スプレッドシート
- 行動ログ
- 現在のコード
- 違和感や相談内容

### 3. 出力
何が出れば循環が進むかを見る。

例:
- 返信文
- Blueprint
- 実装指示
- 入力画面
- 承認フロー
- 判断ログ
- 次の1手

### 4. 構成要素
システムを構成する部品を出す。

人、情報、感情、データ、画面、AI、ファイル、原本、外部サービス、現実行動を分ける。

### 5. 接続
部品同士がどうつながるかを見る。

```md
誰が
何を見て
何を判断し
どこへ渡し
次に何が起きるか
```

### 6. 循環
一度きりの処理ではなく、流れが戻るかを見る。

```md
入力
↓
処理
↓
確認
↓
保存
↓
再利用
↓
改善
```

### 7. 停滞
どこで止まりそうかを見る。

例:
- ユーザーが何を入力すればいいか分からない
- AIが前提を理解できない
- データの正本が分からない
- スマホで面倒
- 手作業が多い
- 権限や個人情報が危ない
- 変更後にテストされない

### 8. 最小介入
最小の部品追加で、最大の循環回復が起きる場所を探す。

いきなり大きなアプリや完全自動化にしない。

まずは、テンプレート、索引、1画面、1ボタン、1つの入力欄、1つのAgent指示で済まないかを見る。

### 9. 将来の拡張性
5年後も壊れにくいかを見る。

確認する問い:
- 原本と正本は分かれているか
- データを取り出せるか
- 外部サービスにロックインされすぎないか
- AIが引き継げるか
- スマホ中心で続けられるか
- 機能を増やしても複雑化しにくいか

### 10. 実行
最後に、実際の成果物へ変換する。

出力は領域によって変わる。

- ホテルメール: 返信文
- 経理: 入力・承認・保存フロー
- アプリ: 実装Blueprint
- 投資: 会社カード
- 健康: 観測ログ
- AI運用: Codex指示
- 人生設計: 次の行動

## 現在最適性チェック
Blueprint Engineは、設計の最後に必ず次を確認する。

```md
現在の前提は何か。
もっと良い構造や技術は出ていないか。
この設計はシンプルさ、保守性、拡張性、循環性の観点で今も最適か。
より少ない部品で同じ目的を達成できないか。
```

技術、API、ライブラリ、料金、連携方法、AIモデルは変わる。

そのため、古い正解を守るのではなく、作業時点の前提を確認して、最小で最適な接続を選ぶ。

## ホテルメールへの応用
ホテルメールも、単なる文章作成ではない。

お客様との信頼循環を回復・強化するための設計物である。

例:

```md
目的: お客様との接続を回復する
入力: クレーム、問い合わせ、予約情報
出力: 安心、理解、返信文、次の行動
構成要素: 感情、事実、期待、対応可能範囲、再発防止
接続: お客様 → 理解 → 安心 → 事実確認 → 対応
循環: 返信 → 納得 → 行動 → 満足 → 評価
停滞: 謝罪前に確認だけする、事実だけ伝える、感情を受け止めない
最小介入: 先に理解と安心を置く
実行: 返信文を作る
```

関連: [[02_PRINCIPLES/Human Communication OS]]

## 経理への応用
```md
目的: レシートから申告数字までの循環を整える
入力: レシート、カード明細、銀行明細、給与、契約
出力: 承認済み経費、仕訳候補、月次集計、申告用数字
構成要素: 人、証憑、支払方法、法人、勘定科目、保存場所、確認者
接続: 入力 → 確認 → 承認 → 保存 → 集計 → 申告
循環: 実務で使う → 詰まりを記録 → 自動化候補へ送る
停滞: 正本不明、入力が面倒、証憑が探せない、AIが前提を知らない
最小介入: まず正本と入力の最小項目を決める
実行: 経理Blueprintを作る
```

## AI設計への応用
```md
目的: 同じ説明を繰り返さず、AIが正しく動く循環を作る
入力: ユーザーの相談、MrBrain、関連ファイル、外部情報
出力: 保存、設計、実装、検証、次の1手
接続: MrBrain → Codex → 実装 → Operation Log → Automation Backlog
停滞: AIごとに理解が分断される、前提が古くなる
最小介入: AGENTS、AI_CONTEXT、Blueprint Engineを読む
```

## 出力ルール
Blueprint Engineを使った出力では、毎回すべての内部思考を長く書かなくてよい。

ただし、重要な設計では次の形式で要約する。

```md
## 目的

## 入力

## 出力

## 接続する部品

## 循環

## 止まりそうな場所

## 最小介入

## 現在最適性チェック

## 次の1手
```
